Features

Full rsync compatibility plus modern performance primitives.

BLAKE3 SIMD Hashing

256-bit cryptographic hashing with automatic AVX-512, AVX2, NEON, and SSE4.1 acceleration. 10-15x faster than MD5 at double the security margin.

Content-Defined Chunking

FastCDC with Gear hashing preserves chunk boundaries across insertions and deletions. A 1-byte edit in a 1 GB file transfers only ~8 KB.

Zero-Copy I/O

Memory-mapped files with madvise(SEQUENTIAL) hints. New files copied via copy_file_range(2) -- data never enters userspace.

Parallel Everything

Directory scanning, chunk hashing, delta computation, and file writing all run on a rayon work-stealing thread pool that saturates all CPU cores.

Atomic Crash Safety

Every file is written to a uniquely-named temp file, then atomically renamed. A crash or power failure never leaves a corrupt destination.

Native TLS 1.3

Built-in transport encryption via rustls. No OpenSSL dependency. Certificate verification. Memory-safe implementation.

rsync-Compatible CLI

Drop-in replacement. Same flags (-avz, --delete, --exclude, --link-dest), same behavior, same muscle memory.

Smart fsync Control

No fsync by default (rsync fsyncs every file). Use --fsync when durability matters. Saves ~15s per 100K small files.

Architecture

A pipelined design where every stage runs in parallel.

Scanner (jwalk, parallel) | v FilterEngine (exclude/include/size filters) | v Per-file decision (rayon thread pool) | +-- New file -------> Applier::copy_new() | (copy_file_range / atomic tmp) | +-- Skip -----------> mtime+size / --size-only / --update | +-- Update ----------> Hasher (BLAKE3 + mmap + madvise) | | | v | DeltaEngine (FastCDC chunks) | | | v | Applier::apply() (atomic tmp + rename) | +-- Delete ----------> --delete / --max-delete safety limit

About

I have been using rsync for over a decade. It is one of the most important tools in the Unix ecosystem -- reliable, ubiquitous, and battle-tested.

But rsync was designed in 1996 for single-core CPUs, spinning hard drives, and 10 Mbps networks. Modern hardware looks nothing like that:

  • CPUs have 16-64 cores, but rsync uses exactly one.
  • NVMe SSDs can sustain 7 GB/s reads, but rsync tops out at ~500 MB/s.
  • Networks are 10-100 Gbps, but rsync cannot saturate even 1 Gbps for many small files.
  • MD5 is cryptographically broken with known collision attacks.

rsync cannot be fixed incrementally. Its architecture is fundamentally single-threaded, and its C codebase makes memory-safe parallelism impractical.

resync-rs is a ground-up redesign: every stage is parallel, I/O is zero-copy, hashing is BLAKE3 with SIMD, and the whole thing is written in safe Rust. Same flags, same behavior, 10-90x faster.

resync-rs stands on the shoulders of these projects:

  • rsync by Andrew Tridgell and Paul Mackerras -- the original delta-sync algorithm that defined a generation of tools.
  • BLAKE3 by Jack O'Connor et al. -- a parallel, SIMD-accelerated hash function.
  • FastCDC by Wen Xia et al. -- the Gear-hash content-defined chunking algorithm.
  • rayon -- Rust's data-parallelism library for safe, work-stealing concurrency.
  • zstd by Yann Collet -- compression at 10x the speed of zlib.
  • jwalk -- parallel directory traversal for Rust.

Getting Started

# Install from crates.io cargo install resync-rs # Or build from source git clone https://github.com/vedLinuxian/resync-rs.git cd resync-rs cargo install --path . # Basic sync resync -avz /data/source/ /data/backup/ # With progress and stats resync -avzP --stats /data/source/ /data/backup/ # Incremental backup with hardlinks resync -aH --link-dest /backups/yesterday /src/ /backups/today/ # Network push with TLS resync push -avz --tls /src/ server:2377:/dst/
